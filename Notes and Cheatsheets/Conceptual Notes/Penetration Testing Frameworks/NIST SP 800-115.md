# NIST SP 800-115 — Technical Guide to Information Security Testing and Assessment

Published by the National Institute of Standards and Technology. A foundational framework for systematically evaluating security posture of information systems. Originally for U.S. federal context but widely adopted in private sector.

**Key distinction:** Not just a pentest framework — covers the full spectrum from document reviews and log analysis to vulnerability scanning and full penetration tests. Penetration testing is one technique among several.

---

## 3 Core Objectives

1. **Identify vulnerabilities** — in systems, networks, and applications
2. **Validate security controls** — test whether they perform as expected under adversarial conditions
3. **Assess exploitability** — simulate real-world attacks to determine if weaknesses can actually be leveraged

---

## The 3 Phases

### Phase 1 — Planning
Define objectives, scope, and rules of engagement before any testing begins.

- What's in scope vs out of scope
- Communication protocols — who gets notified if critical vuln found mid-test
- Permitted testing hours
- Emergency stop conditions
- Output: **formal test plan signed off by all stakeholders**

---

### Phase 2 — Execution
Active testing. Four technique categories, ordered passive to active, low-impact to high-impact:

| Technique | Description |
|---|---|
| **Review** | Examine documentation, policies, configs, firewall rules, ACLs |
| **Target Identification and Analysis** | Discover and fingerprint live hosts, open ports, running services |
| **Target Vulnerability Validation** | Confirm identified weaknesses are real — manually verify, not just scanner output |
| **Penetration Testing** | Simulate adversarial attacks — chain vulnerabilities to demonstrate real-world impact |

**Important:** Not every engagement needs to reach the penetration testing stage. Sometimes review and vulnerability validation are sufficient.

---

### Phase 3 — Post-Testing
Analyze results, prioritize risks, deliver actionable remediation.

- Categorize findings by severity
- Map each finding to the specific control it bypasses
- Provide concrete remediation steps — not just "you have SQL injection" but which parameter, what data is at risk, and how to fix it
- Findings must be **actionable**

---

## Strengths and Weaknesses

| Strengths | Weaknesses |
|---|---|
| Flexible — adapts to data centers, cloud, IoT | Guidance only — not regulation, no enforcement |
| Institutional credibility — recognized by government and defense | Doesn't mandate audit frequencies like PCI DSS does |
| Defines technique categories not specific tools | Requires broad skill set from testers |
| Promotes standardization across teams | |

---

## When It's Used
- Government agencies and federal contractors
- Organizations needing assessments defensible to auditors
- Regulated industries that value structured, repeatable processes
- Anywhere NIST recognition carries weight

---

## Framework Comparison

| Framework | Type | Best For |
|---|---|---|
| NIST SP 800-115 | Guidance | Government, federal contractors, regulated industries |
| OWASP WSTG | Web app testing | Web application penetration testing |
| OSSTMM | Scientific/quantitative | Repeatable, auditable security measurements |
| PCI DSS | Regulation | Payment card industry — mandates annual pentests |
